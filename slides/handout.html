The hardest test suite I ever built

A pytest case study — Grzegorz Kocjan

The full talk in written form. Talk page & code: belazy.dev/talks/the-hardest-test-suite-i-ever-built. Runnable code for every snippet below.

The thesis: the hardest test suite is made of the simplest tests. The difficulty doesn't live in the assertions — it lives in the architecture around them. This is the story of building integration tests for a computer-vision system that watched real cars, on real cameras, 24 hours a day.

Opener — the simplest tests in the world 1.5′

Hook

How do you imagine the integration tests for a really complicated system? Mine was computer vision: real cars, real cameras, running twenty-four hours a day — and right about 85% of the time, by design.

Picture yours. The most complicated system you've ever worked on. How would you write integration tests for that?

slide — the color test, two lines

Here's mine. This is it. This is enough. Twenty-one tests, most a few lines long — the shortest, a single assert. The simplest tests I ever wrote. And together: the hardest test suite I ever built. In the next thirty minutes, I'll show you why both halves are true.

The world before — our regression detector was a human 1.5′

Back to 2021. The system was already live: a computer-vision pipeline watching parking garages — cameras at the entrance, a model reading plates, colors, makes. It had run in production for seventeen months. Integration tests: zero. Unit tests: ninety-six. Tests that could catch a bug: still zero.

So here's how we learned about regressions. We'd change something, deploy, and a day or two later a phone would ring — a garage attendant saying "the system is much worse today." That was our regression detection. A human. Noticing. Calling.

When I say the system was right "about 85% of the time" — that's not a measurement, it's a guesstimate. Before the suite, there was nothing to measure with.

slide — three doors: unit tests and acceptance tests locked, integration tests ajar

Now, why? The code was written by computer-vision scientists — brilliant people, not software engineers — with legacy architecture: global state everywhere, nothing to isolate. Unit tests, acceptance tests? Not an option. Integration tests weren't a philosophy — they were the only door that wasn't locked.

Challenge 1 — first, make it run at all 3.5′

Arrange = start an entire system

The first challenge had nothing to do with assertions. In every test there's that boring part at the top — the arrange. Here, "arrange" meant: start an entire computer-vision system. Services, a database, a model that wants a GPU — and a camera.

We made pytest do it. There's a plugin, lovely-pytest-docker, that lets a fixture bring up a whole Docker Compose stack. The suite doesn't assume the system is running — a fixture starts it: containers up, docker compose up in one line. It doesn't guess with a sleep — it waits until the system is actually responsive, and hands the test a ready URL. Want a different garage, a different scenario? That's just a different JSON file — recordings and their configuration are data, not code. And the arrange is ready — always, automatically.

slide — the web_service fixture that brings up docker-compose (snippet 09)

But the real trick was the camera. You can't test against a live camera — live cars don't reproduce. So instead there's cam_mock: a service that replays frames recorded from the real cameras, as a live stream, at real speed. The system doesn't know it's watching the past. It thinks it's Wednesday afternoon.

And then the test… waits for the recording to be started. When it is, the system is alive, watching a recording where we already know every car, because we put them there. That is the arrange. The system is up, it thinks it's Wednesday afternoon — and a black Honda just pulled in.

Was it fast? No. This is not a suite you run from a pre-commit hook. It runs on a self-hosted GPU runner. A reasonable test suite has a timeout of ten minutes; this one is set to timeout-minutes: 2880. That's forty-eight hours. Hours, not minutes — and with recordings up to twenty-four hours long, even two days is sometimes barely enough. It's an industrial installation, not a unit test.

If you steal one thing here: your system doesn't have to be small to be testable. It has to be startable. Build your services so they start locally with one command, from day one. Record reality once, replay it forever — and suddenly the most untestable system you own has an arrange.

Challenge 2 — thirty seconds, not ten minutes 1′ + 2.5′ + 1.5′ + 3.5′

The goal

So — look how far we've got. The system starts itself, with one command. The recording plays, cars drive in. The tests run against it. Time to assert. Great, right? But…

slide — tally: the system starts itself ✓ · the recording plays ✓ · the tests run ✓ · but…

The first version we built did the obvious thing: play the whole recording, collect everything, check it at the end. It worked — but a recording runs ten minutes, and some run twenty-four hours. So when the first car got a wrong plate in the first thirty seconds, we found out ten minutes later.

slide — "10 minutes later…" / "24 hours later…"

Every debugging cycle cost a full playback. The goal: feedback in thirty seconds, not ten minutes. Getting there took two mechanisms — and the first wasn't enough.

Solution 1 — break the monolith

First we broke one giant end-of-recording check into many tiny tests, one assertion each: plate, color, direction — for every car. In such cases you would normally use @pytest.mark.parametrize — we started there too. But parametrize is a decorator: its values are frozen the moment Python imports the file — a static list. We wanted to choose which recordings to run at runtime, and to feed two fixtures at two different scopes (one per car, one per recording). A decorator can't do that.

So we went one level down — into pytest's basement — to the hook parametrize itself is built on: pytest_generate_tests. Our cases live in JSON; the hook reads them into pydantic models. Does the test want a vehicle_case? Parametrize it with the loaded cases — indirect=True just means the JSON entry arrives as a ready-made fixture, not a raw value. The second block is the same move at a different scope: recording_case — once per whole recording, not per car. A test picks its world by its signature alone — vehicle_case or recording_case — no decorators in the test file, not one. Same file, two scopes, chosen by signature. This helped with legibility, not speed: a failure now said exactly this car, this plate, wrong.

slide — pytest's basement (photo: "Dunkle Kellertreppe" by ThoBel-0043, Wikimedia Commons, CC BY-SA 4.0) + double parametrization (snippet 02)

The problem — pytest runs in the wrong order

By default pytest runs test by test: test_color for every car, then test_plate for every car — ordered by function, not by car. To check the last car's color, the recording has to reach the last car — so to finish, you watch the whole thing. We'd made the report beautiful and left the feedback exactly as slow as before.

slide — default collection order (before)

Solution 2 — reorder, and watch it live

To reorder, you need pytest's startup sequence. First it configures. Then it collects — where pytest_generate_tests built our items. Then, after everything is collected but before a single test runs, one last hook: pytest_collection_modifyitems, which hands you the complete list and says: rearrange it however you like.

slide — pytest lifecycle: configure → collect → modifyitems → run

Two hooks, two phases: generate dynamically, then reorder. We group by recording; within a recording, all of a car's checks fire as that car appears, then the next car. Live tests first; end tests (statistics) last. The boundary is an empty fixture — recording_stopped has no body; it exists so a test can say "I run after the movie ends," by its signature.

slide — live/end orchestration (snippet 03)

One heresy: some of these items aren't tests. test_start_recording asserts nothing — it starts the recording. Every tutorial says a test must assert. Half of this suite disagrees — and in exchange, every lifecycle step shows up in the report with its own timing, for free.

Put it together and something beautiful happens in your terminal: the recording starts and the tests come alive. A car enters, its checks light up; next car — green, green, green; a color warning — yellow, noted. And notice the first and the last line of the output: test_start_recording opens the show, test_wait_for_recording_stop closes it — the lifecycle phases sit right there, like every other item. You're watching the system be judged, car by car, as it happens. A broken deploy fails in the first minute — not after twenty-four hours.

Challenge 3 — the spectrum of trust 1.5′ + 3.5′

The problem — the world where count == 1 dies

For years people called me "the testing guy." My assertions looked like this: add a user, assert the count is exactly one. Two plus two is four, every time. The world where correct means equal. So I asserted everything — plate, color, make, exactly — and the tests failed constantly. Not because the system was broken; it was doing its job about 85% of the time. They failed because reality doesn't come out to exactly four. The hardest part of the project wasn't Docker or hooks — it was unlearning count == 1.

The solution — a spectrum of trust

Remember my very first slide, the color test — assert color == expected? I lied. In the real repository that test asserts nothing. It's warnings.warn("Wrong color"). It cannot fail the build.

slide — "I lied.": assert → warnings.warn

That wasn't laziness — it was the biggest shift in how I think about correctness. Expecting "silver" doesn't mean the model must say "silver": at night or on a filthy car, silver reads grey, and that's fine — color was only ever a hint. Even the plate didn't have to match 100% — a real business requirement the tests had to encode. So I sorted every check into a spectrum of trust — four levels.

At the top — must be exactly right. Direction (in or out — there's no "about"), timing (within a couple seconds), and the plate (with a human tolerance: edit distance of one). Hard assert.

Below — only true-ish. Color, make, model. I don't assert them, I warn. Fail the build every time a dirty silver car reads "grey" and the build is never green — and I'd learn nothing.

slide — the ladder: 4 levels of trust (rungs 3–4 shown as "?")

Why this spectrum? I only understood it standing in a garage. You drive to a booth; a camera reads your plate; an attendant hands you a ticket — plate, color, make, time in — and parks your car. The plate is the bill: it ties money to the car, so it has to be right. Color and make are just hints so a human finds your car in a lot of four hundred; nobody ever corrected those. My spectrum wasn't a clever pattern — it was the business, copied into assertions. Hard-assert what the company bills for. Warn about what only helps a human. Measure the rest.

slide — valet ticket · real git diffs (color→warn Jul 30, make/model→warn Oct 4)

I didn't get there in one go — you can watch it in the git history: a July diff turns the color assertion into a warning; in October make and model follow. The suite learned to trust less, one assert at a time. And once you stop asserting, one thing becomes essential — something that sounds trivial and absolutely is not: counting how often we were right. And counting is where the next challenge was waiting.

Challenge 4 — two cars, same second 45″ + 3.5′

The problem

For a while life was good — until two tests went dead red on an ordinary recording, with both cars detected perfectly. Some garages have wide entrances: two cars roll in side by side, same second. I'd assumed the system reports cars in a fixed order. It doesn't — a classic race condition, the same kind you can hit in a regular REST project: one busy CPU thread, and the order flips. Two tests failed.

The solution — match, don't compare

Instead of comparing sequences, we match — twice: once live, once at the end. Live, when a test asks "dude, where is my car?", a fixture first_sighting escalates: find my plate allowing one wrong character; nothing? Loosen to two; still nothing? Nearest unclaimed event in time — and it keeps trying as long as the recording runs. Whatever it finds it claims, so no other car grabs it.

slide — escalating search (snippet 04)

Like an airport carousel: three suitcases circling, you take yours, check it, move on; the others someone else will claim. Each test waits for its car and leaves the rest on the belt.

slide — carousel (photo: "baggage claim" by bradleygee, Flickr, CC BY 2.0)

After the recording ends, the real matching runs — cars matchmaking. We didn't invent it — we imported scipy.optimize.linear_sum_assignment, the Hungarian algorithm (1955): a matchmaking service pairing expected cars with reported events at the cheapest total cost. All the prices live in a cost matrix — one row per expected car, one column per reported event — and each cell is computed by _weight, essentially one line: wrong characters in the plate, plus delay in milliseconds over ten thousand — an exchange rate where ten seconds late costs one typo. Some pairings are simply forbidden (wrong direction, plate too different) — priced at 10¹².

slide — Cars Matchmaking: cost matrix + Hungarian weight (snippet 05)

Whatever can't be paired becomes the numbers: expected with no event → misses; event with no car → extras (the ghosts); matched but too slow → late. And — remember the four levels of the spectrum? Here is the third one: per recording — statistics for a single recording. One test per recording, three soft checks — that's the pytest-check plugin (soft on purpose — a failing miss must not hide the extras) — every number landing in a shared results bag, a fixture from pytest-harvest that lets us share data between tests; the report comes later.

slide — the ladder, rung 3 "per recording" → level 3: misses/extras/late (snippet 07)

Before the last level — remember the warnings? One wrong color is a warning, fine. But if the model started getting every color slightly wrong, every plate a little off — each one is just a warning, and the build stays green. I'd never notice the system quietly falling apart. A warning catches the single mistake. It cannot catch the trend. So how do you catch the trend?

slide — "Remember the warnings?" · "But a warning cannot catch the trend."

The answer is level four — my favorite test in the whole suite, because it tests no car at all. It runs last, reads the accumulated statistics from the entire run, and asserts one number: the percentage. The bar started exactly where the system was — 85%. When the system got better, we raised it to 90% — one number, in one place, and the whole suite demands more from every deploy. One wrong color is a warning; a trend of wrong colors fails the build. That's how you catch a system quietly falling apart.

slide — the ladder, rung 4 "aggregate" → level 4: summary report (snippet 08)

Zoom out — what we actually built 3′

The test file is 401 lines, 21 tests. The conftest behind it: 744 lines — almost twice as much infrastructure as tests, on purpose. Through double parametrization, that file explodes into 203 recordings, 2,713 expected events — about 39,000 test items. From four pages of code.

Day to day, that complexity is invisible. A new production bug? Cut the recording, drop in a JSON — a new regression case, zero code. A new field? One test, two lines. It's well hidden — you only meet it if you open the conftest, and most days nobody does. The loop it bought us: bug reported → pull the recording → JSON case → fix → smoke tests → long run overnight → next day it ships. Remember the world before? Deploy and wait for the phone to ring. That phone call became a red test on my screen.

How long did it take to build all this — not run it, build it: the docker setup, the hooks, the matching, the reports? Four months. And the clumsy first version — the one where we waited ten minutes for every answer — was already better than anything we had. We built the smallest thing that gave feedback and we continue improving it — including about 40,000 reference values reviewed and corrected by hand, because sometimes the test is wrong, and fixing the test is part of the work.

The take-home — push the Act into a fixture ~2.5′

Most of you will never build something this elaborate — and that's fine. But there's a smaller version I use almost every week. The simplest test: Arrange, Act, Assert, all in the body. When it's simple, that's perfect — you open the test and see everything.

slide — AAA in the body (snippet 10 / test_1)

When the arrange gets heavy, we all know the move: push it into a fixture.

slide — Arrange → fixture (snippet 10 / test_2)

Here's the one fewer reach for: sometimes the Act is expensive — thirty seconds, a big result. One test with twenty asserts hides nineteen behind the first failure; twenty tests pay the thirty seconds twenty times. So push the Act into a module-scoped fixture — it runs once — and let each tiny test pull the result and check one thing. Now the expensive work happens once, and if assert one and assert twenty both fail, you see both. That's the whole principle behind everything here: hide the complexity in the fixtures, keep the tests down to a few lines.

slide — Act → fixture, scope=module (snippet 10 / test_3)

Ending — simple was the result 1.5′

Remember the first slide? Both halves are true. The tests really are that simple — one assert, two lines. And the suite really was the hardest I ever built — because simple was not the starting point. Simple was the result. Four months of work, so that adding a case is dropping in a JSON, adding a check is two lines, and a failing test reads like a sentence.

It's easy to write complicated code that does magical things; it's much harder to get the same result from code that looks simple. Pascal apologized in a letter: "I made this longer only because I had no time to make it shorter." Writing short takes longer. Writing simple takes skill — and if you can't yet, that's not failure, that's a training plan. You have an advantage I didn't in 2021: with AI, iteration is nearly free. Take something you've already tested well — that's your safety net — and try the same problem again, simpler. And again.

The simplest tests I ever wrote. The hardest suite I ever built. Go build yours.
