Why not just @parametrize?

import pytest

CARS = ["car1", "car2", "car3", "car4", "car5"]

@pytest.mark.parametrize("car", CARS)

✗ static — frozen the moment Python imports the file. Can't read runtime config.

[⏱ 7:23] First, we broke that one giant end-of-recording check into many tiny tests — one assertion each. One for the plate. One for the color. One for the direction. Then we ran each of them for every car. Now, in such cases you would normally use @pytest.mark.parametrize. [CLICK] We started there too. But parametrize is a decorator. [CLICK] Its values are frozen the moment Python imports the file — a static list, sitting above the function. [CLICK] We didn't want a static list. We wanted to choose which recordings to run at runtime — from a cases file, from a command-line flag. And we wanted to feed two different fixtures at two different scopes: one per car, one per recording. A decorator can't reach runtime config like that.

Group by recording · live first, end last

@pytest.fixture
def recording_stopped():
    pass  # empty fixture = a phase marker: "I run after the movie ends"

def pytest_collection_modifyitems(session, config, items):
    grouped = defaultdict(lambda: {"live": [], "end": []})
    for item in items:
        case = item.callspec.params.get("vehicle_case") \
            or item.callspec.params.get("recording_case")
        phase = "end" if "recording_stopped" in item.fixturenames else "live"
        grouped[case.recording][phase].append(item)

    items[:] = chain(*[[*g["live"], *g["end"]] for g in grouped.values()])
[⏱ 11:38] [CLICK] Here's the hook itself. [CLICK] We group everything by recording. [CLICK] Live tests, asserting during playback, goes first - to live bucket; end tests, like statistics, last. [CLICK] And the boundary between the phases is marked by an empty fixture — recording_stopped has no body at all. It exists so a test can say "I run after the movie ends," again just by its signature.

Cars Matchmaking

def _weight(expected, received):
    if expected.direction != received.direction:
        return MAX_WEIGHT                       # irreconcilable

    distance = edit_distance(expected.plate, received.last_plate)
    if distance > MAX_EDIT_DISTANCE:
        return MAX_WEIGHT

    lateness = max(0, abs(received.frame_time - expected.appear_at) - 2_000)
    return distance + lateness / 10_000         # 10 seconds late = 1 typo

cost_matrix = [[_weight(e, r) for r in received] for e in expected]
rows, cols = linear_sum_assignment(cost_matrix)   # Hungarian algorithm (scipy)

10 seconds late = 1 typo. MAX_WEIGHT = simply forbidden.

[⏱ 22:04] Then, after the recording ends, the real matching runs — the one that produces statistics. And here we didn't invent anything. [CLICK — the code appears] We imported it: scipy.optimize.linear_sum_assignment. The Hungarian algorithm — born 1955, and it's essentially a matchmaking service: expected cars on one side, reported events on the other; every possible pairing gets a price; the algorithm finds the cheapest total marriage. The interesting part is the price. [CLICK] All the prices live in a cost matrix — one row per expected car, one column per reported event — [CLICK] and each cell gets its _weight, which is essentially one line: the number of wrong characters in the plate, plus the delay in milliseconds divided by ten thousand. Which quietly sets an exchange rate: ten seconds of lateness costs the same as one typo. [CLICK] And some pairings are simply forbidden — priced out entirely — wrong direction, plate too different — The algorithm can marry anyone, but not those two. [CLICK]
expected, no event  →  misses
event, no car  →  extras (the ghosts)
matched, too slow  →  late
[⏱ 23:23] Whatever the algorithm cannot pair becomes one of two numbers. [CLICK] Expected cars with no event: misses. [CLICK] Events but no expected car: extras — the ghosts. [CLICK] Pairs matched, but too slow: late.

The one fewer reach for: Act → fixture

@pytest.fixture(scope="module")
def frame():                          # Arrange
    return load_frame("bha-215.png")

@pytest.fixture(scope="module")
def sighting(frame, recognize_spy):   # Act — runs once per module
    recognize_spy = Mock(wraps=recognize)
    return recognize_spy(frame)

def test_plate(sighting):     
    assert sighting.plate == "BHA-215"

def test_color(sighting):
    assert sighting.color == "navy"

def test_act_ran_once(recognize_spy):
    assert recognize_spy.call_count == 1        # proof: 2 tests, one Act

Hide the complexity in the fixtures. Keep the tests down to a few lines.

[⏱ 28:50] But here's the one fewer people reach for. Sometimes the Act is expensive — a function that runs for thirty seconds and returns a big, rich result. The instinct is one test with twenty assertions. But then the first failing assert hides the other nineteen. And if you split it into twenty tests, you pay the thirty seconds twenty times. [CLICK] So push the Act into a fixture too — module-scoped, so it runs once — [CLICK] Each tiny test just pulls that result [CLICK] and checks one thing. [CLICK] The proof: three tests, and the act ran exactly once. Now the expensive work happens once — and if assert one and assert twenty both fail, you see both, immediately. That's the whole principle, and it's the same one behind everything I showed you today: [CLICK] hide the complexity in the fixtures, keep the tests down to a few lines.
